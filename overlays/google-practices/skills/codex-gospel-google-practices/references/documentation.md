# Google-Practice Documentation

Use this reference for README changes, API docs, comments, runbooks, tutorials, and final reports.

## Reader First

Documentation should help a real reader complete a task or understand a decision.

Write:

- what the reader can do
- when to use it
- prerequisites
- exact commands or examples
- expected result
- failure modes and recovery
- source or decision provenance when it matters

## Style Hierarchy

1. Project-specific docs and terminology.
2. Existing nearby docs.
3. Google Developer Documentation Style Guide.
4. General technical-writing judgment.

Clarity beats rule worship.

## Plain-Language Rigor

For Codex final answers and docs:

- lead with the practical answer
- use normal words for the concept
- include the exact professional anchor only where it changes action
- avoid filler, motivational padding, and repeated caveats
- use examples when the concept is abstract
- do not hide uncertainty

## Comments Versus Docs

- Code comments explain why, invariants, surprising constraints, and source links.
- API docs explain contract, inputs, outputs, errors, examples, and compatibility.
- Runbooks explain symptoms, diagnosis, commands, rollback, and escalation.
- Research docs explain question, method, evidence, uncertainty, and next discriminating check.

Sources: Google Developer Documentation Style Guide and Google Engineering Practices CL description guidance.
